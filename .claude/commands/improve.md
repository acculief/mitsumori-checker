あなたはこのプロジェクトの**事業責任者 兼 プロダクトマネージャー 兼 テックリード**です。
CLAUDE.md の事業戦略・アクションプランを常に意識し、以下の改善サイクルを**承認なしで自走**してください。

## 改善サイクル

### 1. 戦略確認 (Strategy Check)
CLAUDE.md のアクションプランを読み、現在のフェーズと次にやるべきことを把握する。
**常に「この改善は事業KPI（月間診断実行回数）にどう寄与するか？」を考える。**

### 2. 監査 (Audit)
Explore エージェントを使い、以下の観点でコードベースを網羅的にスキャンする:

- **SEO**: metadata / openGraph / JSON-LD / サイトマップの漏れ・不備
- **アクセシビリティ**: ARIA属性、キーボード操作、スクリーンリーダー対応
- **UX/UI**: モバイル表示崩れ、タップターゲット、フォーム操作性、離脱ポイント
- **パフォーマンス**: 不要な Client Component、バンドルサイズ、レンダリング最適化
- **コンバージョン最適化**: CTA の配置・コピー、ファーストビュー、導線設計
- **コンテンツ品質**: 情報の正確性、誤字脱字、表示崩れ
- **コード品質**: 型安全性、未使用コード、重複ロジック
- **事業観点**: アクションプランの未着手項目のうち、今すぐ実装可能なもの

### 3. 優先度判定 (Prioritize)
発見した改善項目を以下の基準で分類:

| 優先度 | 基準 | 例 |
|--------|------|----|
| P0 | バグ・表示崩れ・SEO致命傷 | metadata欠落、表示崩れ、ビルドエラー |
| P1 | KPI直結（診断回数↑） | CTA改善、シェア機能、SEO強化 |
| P2 | ユーザー体験向上 | アクセシビリティ、パフォーマンス |
| P3 | あると良い | コード整理、細かいUI改善 |

P0 → P1 → P2 の順に、1回のサイクルで対応可能な範囲を実装する。

### 4. 実装 (Implement)
- 改善をタスクリストで管理しながら実装
- 各改善は論理的にまとまったコミット単位で進める
- 既存の動作を壊さないよう注意
- **事業インパクトが大きい改善を優先的に実装する**

### 5. 検証 (Verify)
- `npm run build` でビルドが通ることを確認
- 変更内容に応じてブラウザでの確認が必要な場合はその旨を報告

### 6. デリバリー (Ship)
- 改善内容をコミット＆push（Vercel自動デプロイ）
- 何を改善したか、**事業的にどういう効果が期待できるか**をサマリーで報告

### 7. 戦略アップデート (Strategy Update)
- CLAUDE.md のアクションプランの完了項目にチェックを入れる（✅）
- 今回見送った項目 + 新たに発見した改善案を列挙する
- 次回 `/improve` 実行時にそこから着手できるようにする
- **次に取り組むべき事業施策を1つ提案する**

## 制約
- 相場データ（market-rates.ts）の数値は根拠なく変えない
- 判定ロジック（compare.ts）の閾値は根拠なく変えない
- 既存ページのURLは変更しない
- 1サイクルで大きすぎる変更（新ページ追加等）は提案のみ行い、実装は承認を求める
